LingTai TUI follow-up for kernel PR #273

Render full tool_result payloads in the TUI

Jason asked whether PR #273’s nested tool_error metadata would be visible in the terminal UI. Source inspection showed it was LLM-visible but not TUI-visible: the TUI reduced every tool_result event to tool_name → status elapsed_ms. This change makes the TUI render the result payload regardless of shape, with a 10,000-character cap.

Outcome: In Ctrl+O extended view, a failed tool result now shows the normal summary, then tool_error reason / arg keys / guidance when present, then the rendered result. Strings, scalars, arrays, objects, and spill manifests all render in a deterministic form.

Problem

Kernel PR #273 adds model-visible recovery metadata under result.tool_error, but the current TUI session ingestion path threw the payload away when producing the chat stream body:

[tool_result] system → error 12ms

That meant an operator using the TUI could not see why the tool failed or what recovery guidance the model received.

tui/internal/fs/session.goReplaced the one-line tool_result body builder with formatToolResultEvent. It renders any result form; nested tool_error is surfaced first; long result text is capped at 10,000 runes.

Rendered shape

[tool_result] system → error 12ms
  tool_error: system failed during tool_returned_error: event_id is stale
  arg_keys: action, event_id
  guidance:
  - Do not blindly retry the same tool call unchanged.
  - If the failure depends on mutable external state, read the current state before retrying.
  result: {
    "status": "error",
    "message": "event_id is stale",
    "tool_error": { ... }
  }

Cap behavior

The cap applies to the rendered result: text, not to the concise summary lines. It is rune-based, so non-ASCII text is not split mid-byte. If the result exceeds 10,000 characters, the TUI appends:

… truncated to 10000 chars

Existing kernel spill manifests are rendered as their manifest JSON. The TUI does not read sidecar files; that keeps the UI path read-only and avoids unexpectedly loading huge artifacts.

Boundary

This TUI PR does not change the kernel’s tool-result semantics. Kernel PR #273 remains the source of the nested tool_error payload; this PR only preserves and displays that information in the terminal UI.
