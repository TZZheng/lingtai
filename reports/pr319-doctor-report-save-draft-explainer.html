Issue 237: one-step /doctor diagnostic report bundle

Explainer for the TUI change that lets users save a redacted GitHub-ready doctor report from the interactive doctor screen.

TL;DR

This PR adds the first interactive report path for issue #237: after /doctor finishes, the footer offers [r] save report. Pressing r writes a redacted bundle under the global LingTai reports directory without rerunning doctor or mutating the install again.

  • Done: in-memory doctor draft, explicit save action, redacted markdown/JSON/log artifacts, tests, and real temp-project smoke validation.
  • Kept stable: shell lingtai-tui doctor behavior and existing esc handling.
  • Deferred: a shell-first doctor report --output command can be a follow-up if needed.

Baseline

Before this change, /doctor could repair/check the local install and display diagnostic output, but a user filing a GitHub bug report still had to manually copy terminal text and redact private data. The issue asks for a one-step diagnostic bundle that is safe to attach to bug reports.

What changed

AreaChange
internal/doctorreportNew report writer builds report.md, metadata.json, redaction.json, and optional redacted log tail from a doctor draft.
DoctorModelCaptures the finished doctor output as an in-memory draft and exposes [r] save report only when a draft is available and not yet saved.

TestsAdded unit coverage for artifact writing/redaction and TUI behavior including no-rerun save, repeated save, and save failure surfacing.

Risks and decisions

  • Why in-memory draft? It avoids a second doctor pass. This matters because doctor can repair/update the runtime and should not be repeated just to save a report.
  • Why explicit r? No artifact is written unless the user asks for it, matching the issue’s bug-report workflow without surprising users.
  • Why report writer owns redaction? Centralizing redaction keeps future report surfaces from duplicating ad-hoc filtering logic.
  • Remaining risk: Redaction is conservative and covered for common token/path/log shapes, but future diagnostics may add new sensitive fields and should route through doctorreport rather than bypassing it.

  3. When new diagnostic fields are added later, add redaction tests in internal/doctorreport.
